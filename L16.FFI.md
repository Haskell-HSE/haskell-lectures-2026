# FFI, `ForeignPtr` and unlifted types

```haskell
{-# LANGUAGE MagicHash #-}
{-# LANGUAGE TypeApplications #-}
{-# LANGUAGE UnliftedFFITypes #-}

module L16_FFI where

import Foreign.C.Types (CChar, CDouble, CSize (..))
import Foreign.ForeignPtr (ForeignPtr, newForeignPtr, withForeignPtr)
import Foreign.Ptr (FunPtr, Ptr)
import Foreign.Storable (peek, poke)

import GHC.Base (Double#, Double (..), Int#, Int (..), (+#))
```

In this lecture we bridge two worlds: the safe, abstract realm of Haskell and
the raw, imperative world of C. The Foreign Function Interface (FFI) is
Haskell's toolkit for this bridge. However, crossing this bridge requires
handling memory ownership, pointer safety, and understanding a fundamental
Haskell distinction: lifted vs. unlifted types. We will

1. Call C functions from Haskell using the FFI
2. Safely manage foreign memory with `ForeignPtr` to avoid leaks and
   double-frees.
3. Distinguish between lifted and unlifted types, using unlifted types for
   performance and deterministic resource usage.

## The FFI - calling C from Haskell

The FFI allows us to call external C functions. Let's start with a simple
example: we'll call `malloc` and `free` from C's `stdlib.h`.

```haskell
foreign import ccall safe "stdlib.h malloc" c_malloc :: CSize -> IO (Ptr a)
foreign import ccall safe "stdlib.h free"   c_free   :: Ptr a -> IO ()
```

Declarations above specify foreign functions. The `safe` keyword tells GHC
runtime that the foreign function requires additional runtime bookkeeping. This
is the default when `safe`/`unsafe` keyword is omitted. The tradeoffs are the
following:

* `safe` allows C code to call back Haskell functions. With `unsafe`, such
  behaviour is undefined.
* `unsafe` is a bit faster since it skips the bookkeeping.
* If a `safe` call takes a long time, it only blocks the calling thread
  (in `-threaded` mode). If an `unsafe` call blocks, it freezes the entire
  Haskell RTS. So `safe` is a better choice for potentially long-running
  operations like `malloc`.
* GHC 8.4+ ensures that GC never occurs during `unsafe` calls.

Now, let's use these dAngErOus functions directly. Friendly reminder: manual
`malloc` without matching `free` causes memory leaks; double `free` most likely
would do nothing but can possibly cause a crash.

```haskell
useRawPtr :: IO Int
useRawPtr = do
  ptr <- c_malloc @CChar 100
  poke ptr 42     -- Write 42 to the first byte
  val <- peek ptr -- Read it back
  c_free ptr
  return (fromIntegral val)
```

Can you notice the problems with the code above?

## `ForeignPtr` &mdash; managed foreign memory

`ForeignPtr` is a `Ptr` wrapped with a finalizer &mdash; a function
automatically called when the garbage collector determines the `ForeignPtr` is
unreachable, just like with `Weak`. This brings RAII-like safety to Haskell.

We can wrap our raw `malloc`/`free` pair using `newForeignPtr`:

```haskell
foreign import ccall "stdlib.h &free" c_free_ptr :: FunPtr (Ptr a -> IO ())

mallocForeignPtrBytes :: CSize -> IO (ForeignPtr a)
mallocForeignPtrBytes n = do
  ptr <- c_malloc n
  newForeignPtr c_free_ptr ptr
```

Note that, in order to make it work, we had to add another declaration to make
`free` available as a _function pointer_ stored inside a `ForeignPtr`.

Now, we can use it safely:

```haskell
useForeignPtr :: IO Int
useForeignPtr = do
  fp <- mallocForeignPtrBytes @CChar 100
  withForeignPtr fp $ \ptr -> do
    poke ptr 42
    val <- peek ptr
    return (fromIntegral val)
```

The `ForeignPtr` is guaranteed to stay alive during execution of the block
passed to `withForeignPtr`. And the finalizer runs automatically when `fp` is
GC'd. No explicit `free` needed!

## Custom finalizers

Often, you don't just call `free`. You might need to close a file handle,
release a lock, or call a library-specific cleanup.

Suppose we have a C library with a header file `widget_lib.h`:

```c
typedef struct widget widget;
widget* create_widget();
void destroy_widget(widget*);
```

```Haskell
data Widget = Widget

foreign import ccall "widget_lib.h create_widget" c_create_widget ::
  IO (Ptr Widget)

foreign import ccall "widget_lib.h destroy_widget" c_destroy_widget ::
  Ptr Widget -> IO ()

newtype WidgetPtr = WidgetPtr (ForeignPtr Widget)

createWidget :: IO WidgetPtr
createWidget = do
  ptr <- c_create_widget
  fp <- newForeignPtr c_destroy_widget ptr
  return (WidgetPtr fp)

useWidget :: WidgetPtr -> IO ()
useWidget (WidgetPtr fp) = withForeignPtr fp $ \ptr ->
  putStrLn "Using widget"
```

## `ForeignPtr` internals

`ForeignPtr` is a record containing:

* The raw `Ptr a`
* A finalizer (a pointer to an I/O action)
* An optional "plain" pointer for the finalizer's environment

The GC runs finalizers in a separate thread. This is usually safe, but beware:
finalizers are not guaranteed to run promptly (only when GC occurs).

## Reading stuff off of a `Ptr`

In the examples above, we've stored several `CChar`s inside a `Ptr`. What else
can be stored there? Enter `Storable` class:

```haskell ignore
class Storable a where
   {-# MINIMAL sizeOf, alignment,
               (peek | peekElemOff | peekByteOff),
               (poke | pokeElemOff | pokeByteOff) #-}

   sizeOf      :: a -> Int
   -- ^ Computes the storage requirements (in bytes) of the argument.
   -- The value of the argument is not used.

   alignment   :: a -> Int
   -- ^ Computes the alignment constraint of the argument.  An
   -- alignment constraint @x@ is fulfilled by any address divisible
   -- by @x@. The alignment must be a power of two if this instance
   -- is to be used with 'alloca' or 'allocaArray'.  The value of
   -- the argument is not used.

   peekElemOff :: Ptr a -> Int      -> IO a
   -- ^ Read a value from a memory area regarded as an array
   -- of values of the same kind.  The first argument specifies
   -- the start address of the array and the second the index into
   -- the array (the first element of the array has index
   -- @0@).

   pokeElemOff :: Ptr a -> Int -> a -> IO ()
   -- ^ Write a value to a memory area regarded as an array of
   -- values of the same kind.

   peekByteOff :: Ptr b -> Int      -> IO a
   -- ^ Read a value from a memory location given by a base
   -- address and offset.

   pokeByteOff :: Ptr b -> Int -> a -> IO ()
   -- ^ Write a value to a memory location given by a base
   -- address and offset.

   peek        :: Ptr a      -> IO a
   -- ^ Read a value from the given memory location.
   --
   --  Note that the peek and poke functions might require properly
   --  aligned addresses to function correctly.  This is architecture
   --  dependent; thus, portable code should ensure that when peeking or
   --  poking values of some type @a@, the alignment
   --  constraint for @a@, as given by the function
   --  'alignment' is fulfilled.

   poke        :: Ptr a -> a -> IO ()
   -- ^ Write the given value to the given memory location.
```

Note that there's no builtin magic (except in the instances for builtin C types):
you can write down instances for your own types yourself! Just respect
the laws written out in the docs, please.

## Unlifted types

So far, all of our types (`Int`, `Ptr`, `ForeignPtr`) have been lifted. Lifted
types are heap-allocated, can be thunks (unevaluated expressions), and can be
`undefined`. They are represented by a pointer to a closure.

Unlifted types are different:

* They are not heap-allocated (or are allocated as plain data, not closures).
* They represent values directly (e.g., machine integers, raw bytes).
* They cannot be `undefined` or thunks – they must be fully evaluated.
* They are used for performance and for interacting with low-level primitives.

Another important distinction is that polymorphic Haskell functions are only
polymorphic over _lifted_ types. This limitation is due to polymorphism working
through _erasure_ at runtime: it assumes that each value is represented with an
opaque pointer. For unlifted types this is obviously not the case.

## Examples of unlifted types

* `Int#`: raw machine integer (like C long)
* `Double#`: raw double-precision float
* `Char#`: raw character
* `Addr#`: raw machine address
* `ByteArray#`: contiguous array of bytes

These types live in `GHC.*` modules. Notice the `#` suffix: in order to work
with such types, a `MagicHash` language extension is required.

```haskell
plusUnlifted :: Int# -> Int# -> Int#
plusUnlifted x y = x +# y

liftedAdd :: Int -> Int -> Int
liftedAdd (I# x) (I# y) = I# (plusUnlifted x y)
```

## Why unlifted types matter for FFI

When calling C functions that return simple values (e.g., `int`, `double`), we
don't want the overhead of lifting them into a Haskell heap object. For such
usecases, GHC offers the `UnliftedFFITypes` language extension.

```c
double dot_product(double* a, double* b, int n);
```

Then in Haskell:

```haskell
foreign import ccall "dot_product" c_dot_product ::
  Ptr CDouble -> Ptr CDouble -> Int# -> Double#

dotProduct :: Ptr CDouble -> Ptr CDouble -> Int -> Double
dotProduct a b (I# n) = D# (c_dot_product a b n)
```

The `Int#` argument is passed directly as a machine integer; no heap allocation
occurs. The `Double#` result is returned in a floating-point register. This is
as efficient as calling C from C.

Tip: you can use `inline-c` library to write C code blocks directly inside
Haskell, bypassing having to split source files
(as is the case with this example).

## Warnings & best practices

* `ForeignPtr` finalizers run on GC: if you need deterministic cleanup (e.g.,
  for a lock), use a `bracket` pattern or the `ResourceT` monad.
* Unlifted types cannot be stored in polymorphic containers: you cannot put an
  `Int#` in a `List` or `Maybe`. You must wrap it (lifting it!) first.
* Don't mix `Ptr` and `ForeignPtr` carelessly: if you extract a `Ptr` from a
  `ForeignPtr` and then drop the `ForeignPtr`, the finalizer might run while you
  still use the `Ptr`. Always use `withForeignPtr`.
* `unsafePerformIO` with `ForeignPtr` is dangerous: unless you understand the GC
  and pointer lifecycles perfectly, avoid it.
